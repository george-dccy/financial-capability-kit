# 模型与模式兼容性说明

## 先说结论

不是所有聊天类大模型应用都能真正读取这个仓库。

“我把仓库地址贴给模型了” 不等于 “模型已经读取了 `registry/*.json` 和对应正文文件”。
真正决定效果的，是当前应用是否支持：

- 克隆仓库
- 读取网页中的文件内容
- 在当前模式下持续受仓库约束

以下说明基于 **2026-04-16** 的当前实测结论，后续可能随产品更新而变化。

## 推荐顺序

| 工具 / 模式 | 当前结论 | 建议 |
|---|---|---|
| `MiniMax 全能模式` | 可以克隆仓库后读取内容并回答，当前效果最好 | 优先推荐 |
| `豆包电脑版超能模式` | 可以读取仓库内容 | 推荐 |
| `DeepSeek + 深度思考 + 关闭智能搜索` | 更容易按仓库内容回答 | 可用 |
| `千问等普通聊天模式` | 容易绕开仓库自由发挥 | 不建议作为主入口 |
| `本地 Agent / 可读本地仓库的 Agent` | 直接读取本地文件最稳定 | 强烈推荐 |

## 各模式说明

### MiniMax

- 当前优先推荐 `全能模式`
- 优点是可以先克隆仓库，再读取具体文件内容
- 更适合作为“仓库优先”的对话入口

### 豆包

- 目前建议只在 **电脑版 `超能模式`** 中使用本仓库 prompt
- 普通聊天模式不保证会真实读取仓库正文
- 如果模型没有明确命中 `registry/*.json` 和相关正文文件，就不要相信它已经按仓库回答

### DeepSeek

- 建议开启 `深度思考`
- 建议关闭 `智能搜索`
- 如果不关闭智能搜索，模型容易被实时搜索结果带跑，而不是优先使用仓库内容

### 千问等普通聊天模式

- 不少聊天应用会把仓库链接当作普通背景信息
- 结果往往是：看似“理解了仓库”，实际上还是按通用金融常识自由发挥
- 这类模式适合闲聊，不适合作为本仓库的主工作入口

### 本地 Agent / 能读仓库的 Agent

- 如果 Agent 能直接 clone 仓库，或读取本地文件，是最稳定的方式
- 这也是当前仓库最优先支持的长期形态

## 使用前如何自检

在正式提问前，建议先让模型完成一次仓库访问校验：

1. 是否成功访问 `registry/skills.json`、`registry/knowledge.json`、`registry/prompts.json`
2. 是否能列出实际读取到的文件路径
3. 是否能基于正文文件给出一句话摘要

如果模型做不到这三步，就不要把它当成“已接入仓库”的模式。

## 推荐写法

可在 prompt 前面加一段简单校验：

```text
在回答前，先判断你是否真的读取到了这个仓库。
如果没有成功读取 `registry/skills.json`、`registry/knowledge.json`、`registry/prompts.json` 和至少 1 个相关正文文件，请直接写“当前无法访问仓库”，不要继续按常识补全回答。
```

## 这份说明的用途

这份说明不是为了比较模型强弱，而是为了避免误导用户：

- 不要把“贴了仓库地址”误认为“模型已经按仓库工作”
- 不要让用户在不支持仓库读取的模式里误以为 prompt 已生效
- 不要把模型自由发挥的答案，误判为仓库内容本身
