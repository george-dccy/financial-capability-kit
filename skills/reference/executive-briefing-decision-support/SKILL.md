# 向上汇报与决策支持判断框架

这是一个 reference skill。它不输出具体汇报内容，而是提供汇报前的判断框架：用什么结构、识别哪些是真正的决策点、预判哪些风险需要提前准备。

## When To Use

- 准备向领导汇报前，犹豫用哪种汇报结构
- 需要识别汇报中的决策点有哪些
- 需要预判领导可能的质疑和风险点
- 需要判断这件事需要领导"知道"还是"拍板"
- 不确定汇报应该结论先行还是背景先行

## Core Output Rule

每次汇报前至少明确：

1. **汇报类型判断**：同步型 vs. 请求型 vs. 说服型
2. **结构选择**：结论先行还是背景先行
3. **决策点列表**：这次汇报需要领导做几个决定
4. **风险预判**：领导可能质疑什么，需要提前准备什么
5. **资源需求**：需要领导给什么（时间、人、钱、权）

## Not For

- 提供具体汇报话术或文字内容
- 替代 `skill.reference.decision-brief-framework` 的结构输出
- 替代 `skill.action.report-decision-brief` 的执行脚本
- 在没有基础事实时硬做判断

## 判断框架

### 框架一：汇报类型判断矩阵

| 类型 | 定义 | 领导角色 | 结构要求 |
|------|------|----------|----------|
| **同步型** | 让领导知道进展/结果 | 被告知 | 简洁摘要 + 关键事实 |
| **请求型** | 需要领导给资源/授权 | 决策者 | 结论先行 + 支撑逻辑 |
| **说服型** | 需要领导改变主意/批准方案 | 被说服 | SCQA 结构 + 风险对应对 |

判断规则：

```
第一步：这件事不汇报会怎样？
  ├── 什么事都不发生 → 同步型（重新考虑是否值得汇报）
  └── 什么事会卡住/有风险 → 进入第二步

第二步：需要领导做什么？
  ├── 给意见/表态 → 同步型（加决策点）
  ├── 给资源/人/时间 → 请求型
  └── 改变原决定/批准方案 → 说服型
```

### 框架二：结构选择判断

**结论先行**（Pyramid Principle）：
- 请求型汇报（要资源、要批准）
- 领导时间紧
- 已有明确建议动作

结构：结论 → 关键事实 → 当前卡点 → 建议动作 → 需要拍板项

**背景先行**（SCQA）：
- 说服型汇报（要改变领导主意）
- 领导对背景不熟悉
- 结论会出乎意料

结构：背景（S） → 冲突（C） → 问题（Q） → 答案（A）

**银行场景改编**：

| 场景 | 推荐结构 | 原因 |
|------|----------|------|
| 季度经营分析汇报 | 背景先行 | 领导需要先建立整体印象 |
| 问题升级要资源 | 结论先行 | 时间紧，领导要快速判断 |
| 方案审批汇报 | SCQA | 需要铺垫逻辑说服 |
| 风险预警汇报 | 结论先行+风险对应对 | 紧迫性优先 |
| 常规工作同步 | 结论先行+简略事实 | 效率优先 |

### 框架三：决策点识别四问

- **Q1：这次汇报要让领导做几个决定？** 1个聚焦打透；2-3个排序主次；3个以上考虑拆分
- **Q2：每个决定的选项是否已收敛到2-3个？** 没收敛先内部讨论
- **Q3：每个决定的风险是否已预判？** 领导同意的最坏结果、拒绝的退路、质疑的备选数据
- **Q4：需要领导给的资源是否明确？** 量化（"2小时+1人"）而非模糊（"需要支持"）

### 框架四：领导质疑预判矩阵

| 质疑类型 | 领导潜台词 | 提前准备 |
|----------|------------|----------|
| 数据质疑 | "这数据哪来的？" | 数据来源页 |
| 逻辑质疑 | "因果关系成立吗？" | 推导过程 |
| 优先级质疑 | "为什么这事排前面？" | 对比说明 |
| 资源质疑 | "值不值得花这么多？" | ROI 或紧迫性说明 |
| 风险质疑 | "有什么风险？" | 风险对应对 |
| 权限质疑 | "不在你范围内吧？" | 授权说明 |

预判流程：列出最强3个质疑点 → 每个准备30秒应答 → 判断哪个可能推翻结论

### 框架五：汇报结构选择流程

```
第一步：判断汇报类型 → 同步型/请求型/说服型
第二步：识别决策点数量 → 1个单线 / 2-3个主次 / 3+考虑拆分
第三步：预判质疑点 → 有强质疑用SCQA / 无强质疑用结论先行
第四步：确认资源需求 → 明确量化与交换条件
```

## 示例

### 示例一：问题升级要资源（请求型+结论先行）

**场景**：某客户项目遇到技术障碍，需要领导协调IT部门支持，预计需要2周时间和1个开发人员。

**判断过程**：
1. 判断类型：不汇报项目会延期 → 请求型
2. 决策点：1个（是否批准IT资源投入）
3. 质疑预判：资源质疑 + 优先级质疑
4. 资源需求：明确量化（2周，1人）

**框架输出**：
- 汇报类型：请求型 → 结论先行
- 决策点：是否批准IT资源投入
- 风险预判：准备项目延期影响说明 + 项目价值和紧迫性说明
- 资源需求：2周 + 1人

### 示例二：方案审批汇报（说服型+SCQA）

**场景**：客户提出非标方案，超出常规授权，需要领导特批。领导倾向保守处理。

**判断过程**：
1. 判断类型：需要改变领导原有判断 → 说服型
2. 决策点：1个（是否批准非标方案）
3. 质疑预判：风险质疑 + 逻辑质疑
4. 结构选择：SCQA（需要铺垫逻辑）

**框架输出**：
- 汇报类型：说服型 → SCQA
- 决策点：是否批准非标方案
- 风险预判：准备风险对应对和止损机制 + 标准方案不适用的说明

### 示例三：风险预警汇报（结论先行+风险对应对）

**场景**：贷后监控发现某客户现金流异常，需要领导关注。

**判断过程**：
1. 判断类型：预警性同步，可能后续需要拍板 → 同步型（带决策点）
2. 决策点：2个（是否提高监控频率 + 是否需要提前介入）
3. 质疑预判：数据质疑 + 行动质疑
4. 结构选择：结论先行+风险对应对

**框架输出**：
- 汇报类型：同步型（带决策点）→ 结论先行
- 决策点：主=提高监控频率，次=提前介入（可延后）
- 风险预判：准备数据来源说明 + "如果不动会怎样"的说明

## 常见误区

### 误区一：把同步型当成请求型汇报

向领导做了5分钟背景介绍，领导问"所以你要我做什么"。同步型不需要领导做决定，混用类型会导致结构错位。汇报前先问自己"不汇报会怎样"，如果什么都不会卡，先考虑值不值得汇报。

### 误区二：决策点超过三个没有排序

一次性列出5个决策事项，没有主次。领导注意力是稀缺资源，超过3个决策点会导致决策疲劳。最多报3个，其余延后或拆分。

### 误区三：没有预判风险就上场

汇报方案只讲正面因素，领导一问风险就卡住。汇报前做"反对意见预演"，列出最强3个质疑点并准备30秒应答。

### 误区四：资源请求模糊化

汇报结束说"希望领导支持"或"需要一些资源"。模糊的资源请求等于没有请求。必须量化（时间、人数、金额、权限范围），如果自己都无法量化，说明还没想清楚。

## Quality Gate

- 是否仍有判断框架而非内容模板
- 是否有银行/金融场景针对性
- 是否与 decision-brief-framework 和 report-decision-brief 边界清楚
- 是否有管理咨询公开方法论来源支撑
