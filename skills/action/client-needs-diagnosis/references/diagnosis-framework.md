# 客户需求诊断框架

## 诊断三步法

### Step 1：信号采集

从以下维度采集已知信号，不要求全部有，越多诊断越准：

| 维度 | 信号类型 | 示例 |
|------|---------|------|
| 业务动作 | 客户在做什么业务动作 | 扩产、采购、项目交付、回款、出口、备货 |
| 资金环节 | 客户提过哪类资金问题 | 到账慢、对账难、垫资大、付款周期长 |
| 外部变化 | 客户近期遇到什么变化 | 政策变化、汇率波动、原材料涨价、订单增加 |
| 客户表达 | 客户直接说过什么 | "现金流紧"、"希望少跑银行"、"想有个方案" |
| 历史行为 | 客户以前怎么处理的 | 民间拆借、他行贷款、拖延付款 |

### Step 2：信号归类

把每个信号归入以下需求类型之一：

- **结算与现金管理**：日常收付款、资金归集、账务管理
- **贸易融资**：采购垫资、销售回款保障、信用证、供应链账期
- **供应链金融**：核心企业上下游融资、确权、担保
- **普惠金融**：小微企业贷款、税务贷款、场景贷款
- **信贷融资**：固定资产贷款、流动资金贷款、项目融资
- **外汇与跨境**：结售汇、跨境结算、保函、汇率避险
- **数字化转型**：银企直连、API 接口、电子票据、供应链平台

### Step 3：置信度判断

| 置信度 | 条件 |
|--------|------|
| 高 | 有 2 个以上不同维度的信号指向同一需求类型 |
| 中 | 有 1-2 个明确信号指向某需求类型 |
| 低 | 信号模糊或指向多个需求类型，无法排除其他可能 |

## 诊断注意事项

- 不要在信号不足时强行定型，待验证项要明确写出
- 主需求和次需求要区分，主需求是当前最值得跟的方向
- 客户表达的"想要"不等于"需要"，要还原到业务动作判断
- 政策变化、宏观波动等外部信号往往触发多种需求叠加
