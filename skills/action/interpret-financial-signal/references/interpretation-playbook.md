# Interpretation Playbook

## 1. 基本顺序

1. 先复述一句“这条信息可能意味着什么”
2. 判断它影响哪些对象：客户、银行、产品、风险、行业、个人能力
3. 判断影响路径：收入、成本、现金流、融资、结算、风险、效率、合规、信任
4. 判断它对当前用户的直接价值：学习、营销、跟进、汇报、沉淀
5. 给出最小可执行动作，而不是泛泛建议

## 2. 事实、推论、建议分层

输出时要自然区分三层：

- 公开事实：材料里已经明确写出的内容，或仓库已有来源能支持的内容
- 专业推论：基于行业、客户和金融工作逻辑做出的判断
- 推进建议：可以实际采取的动作、话术、清单或下一步验证

不要把推论写成事实。
不要把建议写成承诺。

## 3. 常用判断框架

政策类信息：

- 监管想解决什么问题
- 哪些机构或客户行为会被鼓励
- 哪些行为会被约束
- 对客户经理、产品经理、管理者各有什么动作

产品或竞品类信息：

- 服务什么客户
- 解决什么场景痛点
- 依赖什么数据、渠道、风控或生态
- 与本仓库已有知识有什么相似或差异
- 是否值得拆成 knowledge 或 action skill

客户变化类信息：

- 变化发生在经营、财务、组织、股权、订单、项目还是风险侧
- 对现金流、结算、融资、账户、供应链、员工、舆情的影响
- 当前最应该验证哪 1-2 个问题
- 下一次触达要避免什么误判

能力素材类信息：

- 这是不是可复用的工作方法
- 它更像 reference skill、action skill、knowledge 还是 memory
- 是否只适合 private-first
- 是否有公开价值和公开来源

## 4. 常见误区

- 看到新闻就直接写宏观评论，缺少用户工作场景
- 看到产品就直接营销，缺少客户痛点和证据
- 看到竞品就贬低竞品，缺少可学习维度
- 看到客户变化就直接推荐融资，忽略风险和信任
- 看到政策就只引用条文，缺少实际影响路径
