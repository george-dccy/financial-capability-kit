# Knowledge Routing

## 光大产品路由

### 光大交易银行产品目录

- **命中条件**：涉及光大交易银行产品、产品推荐、产品比较、产品名核验或客户适配
- **知识包**：`pack.banks.ceb.transaction-banking.product-catalog`
- **适用场景**：先确认可输出的产品名称范围，避免使用仓库未覆盖或不在当前清单中的名称
- **硬边界**：不在产品目录里的名称不得作为光大交易银行产品输出；`阳光e支付` 不在目录中，不得推荐或解释为光大交易银行产品
- **平台边界**：`阳光供应链云平台` 是对客服务平台/线上化渠道能力，可作为承载平台说明；不能替代具体产品名做推荐

### 结算底座

- **命中条件**：客户卡点在账户、收付、回单、对账、付款流程、授权管理
- **知识包**：`pack.banks.ceb.corporate-settlement.basic-settlement`
- **适用场景**：首次接触需厘清基础结算结构；卡点在收付款执行层面而非融资层面
- **优先级**：当问题停留在"账户怎么开、款怎么收、账怎么对"时，优先命中

### e付通（1+N 保理线上化场景）

- **命中条件**：明确出现 e付通、1+N 保理、核心企业授信支持供应商融资、应收账款转让、线上化确权或相关公开材料
- **知识包**：`pack.banks.ceb.transaction-banking.e-fu-tong`
- **适用场景**：客户处于核心企业-供应商交易链中，且已明确进入 e付通或 1+N 保理线上化讨论
- **行业触发**：行业本身不能触发 e付通；必须有核心企业、供应商、应收账款、1+N 保理或 e付通明确信号
- **优先级**：作为白名单内可选产品，不因“供应链”泛泛命中而默认首推

### 电费通（电费证）

- **命中条件**：出现电费、电费缴纳、电网、电费结算单、国内证、福费廷、电费支付优化、月度缴费
- **知识包**：`pack.banks.ceb.trade-finance.dian-fei-tong`
- **适用场景**：客户有持续、大额电费支出，卡点在缴费节奏与资金占用优化
- **行业触发**：高耗能制造（化工、钢铁、有色金属）、数据中心、大型园区、商业综合体
- **优先级**：当场景明确围绕"电费怎么交、资金怎么安排"时，优先命中

### 阳光供应链云平台（对客服务平台）

- **命中条件**：明确出现阳光供应链云平台、供应链线上化平台、线上确权、对客服务入口、平台承载
- **知识包**：`pack.banks.ceb.transaction-banking.yangguang-supply-chain`
- **适用场景**：解释供应链金融业务如何线上化承载、如何配合 e付通/保理等具体业务使用
- **边界提示**：它是平台/渠道能力，不是具体融资、结算或贸易金融产品；产品推荐仍需回到产品目录白名单

## 通用能力路由

- 客户推进方法、复盘和优先级 -> `pack.common.banker-thinking.top-performer`
- 宏观与经营解释 -> `pack.common.economics.business-basics`
- 需求洞察与异议处理 -> `pack.common.sales.consultative-b2b`
- 信任建立与沟通心理 -> `pack.common.psychology.business-communication`

## 路由优先级

1. **产品目录优先**：涉及光大交易银行产品名时，先读产品目录白名单
2. **场景特异性优先**：产品知识包（结算 / e付通 / 电费通）优先于通用能力包
3. **产品知识包内**：只有明确命中具体产品或场景时，才进入对应产品知识包；不要从行业词直接跳到产品推荐
4. **多包命中**：最多选 1 个产品主包 + 1 个产品备选包 + 1 个通用能力包，避免会谈失焦
5. **通用能力包**：始终按需补入，不作为主推切入口

## 回退逻辑

- 命中供应链/核心企业/应收账款场景但未明确产品 -> 先走产品目录和需求诊断，只输出“供应链金融/保理方向待核验”，不默认走 e付通
- 命中电费/能源缴费场景但未明确产品 -> 进入电费通/电费证场景核验
- 命中基础收付/账户问题 -> 走基础结算
- 无法判断场景类型 -> 走基础结算，并在输出中标注"需进一步厘清客户站位"

## 读取建议

1. 先读取产品主包 `README.md` 与 `modules/*`
2. 再读取产品主包 `faq.md`，用于客户常见问题应答
3. 如有备选产品包，补读其 `README.md` 和 `faq.md`
4. 最后补通用能力包，提升沟通与推进质量

## 多包场景处理

- 命中多个产品包时，按场景特异性选主包，另一个作为备选
- 输出时明确"主推切入口"与"备选切入口"
- e付通与结算同时命中时：若问题明确涉及 e付通/1+N 保理线上化，主包选 e付通；若只是基础收付执行，主包选结算；若只是泛供应链场景，先做需求诊断，不直接给具体产品名
