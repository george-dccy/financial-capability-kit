# 光大银行-e付通知识包

## 适用问题

- e付通在公开资料里更接近什么产品或场景表达
- 它和 1+N 保理、阳光融e链分别是什么关系
- 首次沟通时该先判断什么，哪些话不能先承诺

## 命名规则

本目录路径中的 `e-fu-tong` 只是文件 slug，不是产品英文名。
对用户输出时只能写 `e付通`，不得翻译或改写成 `阳光e支付`、`阳光 e支付` 或其他目录外名称。

## 模块

- `product-positioning.md`
- `sales-discovery-questions.md`
- `customer-faq.md`

## 使用建议（给 Agent）

1. 先读 `product-positioning`，用公开来源把 e付通放回到"核心企业、供应商、应收账款、线上化"的真实语境里。
2. 再读 `sales-discovery-questions`，把客户模糊需求拆成角色、交易基础和线上化条件三类判断。
3. 客户问答阶段优先读 `customer-faq` 与本文件的「常见问题」章节，保持结论稳妥，不把场景判断说成审批结论。
4. 如果需要把知识转成会谈提纲或推进动作，补读 `skills/action/market-corporate-client`。

## 边界

该包只支持公开知识解释与场景理解，不输出审批、授信、费率、时效、系统对接能力或上线承诺。

## 常见问题

### Q1：是否可以把 e付通直接当成普通支付工具来讲

不建议。
当前公开资料更支持把它理解为围绕核心企业、供应商、应收账款和线上化流程的供应链金融场景，而不是一个单点支付按钮。

### Q2：是否可以直接承诺供应商一定不用单独审批

不可以。正式条款需以实际沟通材料为准。
公开材料里有分行层面的 `1+N(e付通)` 表述，但是否适用到当前客户，仍要看正式产品安排、授信结构和分行执行口径。

### Q3：客户问"适不适合我"，怎么答更稳妥

先用场景诊断回答，不直接下结论：
看客户是在核心企业侧还是供应商侧、交易链是否稳定、应收账款是否可确认、线上化协同条件是否具备，再决定是否进入下一步沟通。

### Q4：它和 1+N 保理、阳光融e链是什么关系

公开资料能支持这样的理解：
`1+N` 保理提供基础业务结构，`阳光融e链` 提供线上化服务能力，`e付通` 更接近把这两类能力落到核心企业-供应商场景中的具体模式表达。

### Q5：能不能直接报费率、额度、时效或上线计划

本知识包不提供费率和时效承诺。
具体以正式业务沟通和审批流程为准。

### Q6：第一次会谈最值得达成什么结果

最好不是直接谈价格，而是先把三件事说清：
客户站位、交易基础、下一步要补哪份材料。
如果这三件事没理顺，后面很容易跑偏。
